    小程序CSS/LESS规范

    嘤嘤喵喵子🍑 4/26/2021 代码规范

    # 前言

    1. 一方面，本文档希望定制一套代码风格统一的规范，使大家开发出的代码保持风格一致，更加易于理解与维护。

    2. 另一方面，鉴于微信小程序代码转换支付宝小程序出现的样式未隔离，各种组件与页面同类名样式覆盖问题，希望能通过规范来解决这一问题。

    备注：此文档仅仅是提供给大家的一套开发规范，虽然有些可以通过工具进行检查以及修复，但是还有一些是需要大家自觉遵守执行的，希望大家都能认真对待。

    # 强制执行

    1. 禁止使用标签选择器，page 除外 标签选择器，尤其是在顶层使用的标签选择器，选择范围过大，且在小程序组件内，会超出组件样式隔离的效果，影响到页面样式。【更何况，我们现在也都知道在支付宝小程序内根本没有组件与页面的样式隔离】

    TODO: 后续开发过程中，遇到有标签选择器时，尽量改为 .class 选择器，如果是像活动页那样影响范围极大，不好改动的，可以暂时不要改动。

    但在新开文件，复制早期内容时，一定要杜绝使用标签选择器。

    1. 禁止嵌套的ID选择器【最好不要使用ID选择器】 从性能上来说，ID选择器确实优于Class选择器，但是这部分性能提升并没有多大。

    除此以外，ID选择器相比Class选择器并没有什么其他的高级功能，反而因为ID的唯一性，使其缺失了复用性。

    而且ID选择器提升了一个等级的优先级，使得后续的样式覆盖变得不易处理：即使你用无限个Class选择器，也无法覆盖一个ID选择器。

    1. 组件必须有一个scope 由于支付宝小程序，页面与组件缺少样式隔离，所以我们需要手动的添加隔离方式。

    给每个组件的最外层元素添加一个以组件为名的Class；

    然后CSS中的样式全部嵌套在这个Class的选择中。

    示例：

    <!-- component.wxml -->
  <view class="component-container">
    <view class="inner-content">...</view>
  </view>
  // component.wxss
  .component-container {
    .inner-content {
      // ...
    }
  }

    1. 定位的z-index设置不要过高 正常的定位层级在20以内基本能够满足大部分情况。部分确实需要高层级的，50以内也完全足够用了，像一些设置999，9999，10000，等值都是无谓的拔高层级设置。

    如此也会导致后续定位层级步步拔高，层级之间难以维护。

    我们之前也有过这方面的规范文档写在项目中，这里再次重申一遍，希望大家能够牢记。

    目前我们参考bootstrap定义规范结合我们项目给出层级规范定义，z-index 大小按照从上往下，从左往右依次增大

    页面一般元素 10 以内

    sticky 元素，悬浮元素 11-19

    footer （置底fixed元素），searchbox （置顶搜索框），sidebar（侧栏） 21-29

    tooltip 提示框（错误信息等） 91-99

    modal 弹窗（定义：只能操作弹窗中的内容） 100以上，虽说是100以上，但并不是可以无限大的，应该尽量小的偏向100，能设置100，就不要设置101，这种情况。

    如果真的因特殊原因，需要设置更高的 z-index ，需要在上一行添加备注，写明为什么要设置这么高的层级。

    # 推荐规范

    OOCSS和BEM OOCSS (全称 Object Oriented CSS, 即面向对象的CSS)，其思想就是鼓励你把样式表看作“对象”的集合：创建可重用性、可重复性的代码段让你可以在整个网站中多次使用。

    # 参考资料

    Nicole Sullivan 的 OOCSS wiki Smashing Magazine 的 Introduction to OOCSS BEM (全称 Block-Element-Modifier, 即块-元素-修饰符)，这是一种用于 HTML 和 CSS 类名的命名约定。

    BEM 最初是由 Yandex 提出的，要知道他们拥有巨大的代码库和可伸缩性，BEM 就是为此而生的，并且可以作为一套遵循 OOCSS 的参考指导规范。

    CSS Trick 的 BEM 101 Harry Roberts 的 introduction to BEM BEM类似于模块化，首先你需要根据 HTML 的结构，将其中一些有内在联系的嵌套元素整体视为一个块(Block)，而内部的每一部分，就是这个块的一个元素(Element)。修饰符(Modifier)则是用来区分一些不同块/元素的不同状态。

    整体的命名规范为：block__element--modifier。

    最外层的块为 block，内部的元素为 block__element，如果有其他的修饰样式，则添加 --modifier 的后缀。

    比如：

    <view class="warning-modal warning-modal--opacity">
  <view class="warning-modal__title">Warning</view>
  <view class="warning-modal__content">You can't do this</view>
  <button class="warning-modal__button warning-modal__button--primary">OK</button>
</view>

    以上示例中，warning-modal 即是块(Block)。里面有三部分: 标题、内容与按钮，这就是元素(Element)。最后 warning-modal--opacity 和 warning-modal__button--primary 后面的部分即是修饰符(Modifier)，一个表示给整个 warning-modal 块加上透明度，一个表示给 warning-modal 的按钮设置为主按钮的样式。

    如此命名的一个好处是，很容易将各个部分与块的关系表示出来，容易理解，便于维护。

    还有一个好处是，使用 PreCSS 写样式时，通过嵌套写法也更方便。

    比如上面的结构，使用 Less 来写样式的话，可以用如下方式：

    .warning-modal {
  // ...

  &--opacity {
    // ...
  }

  &__title {
    // ...
  }

  &__content {
    // ...
  }

  &__button {
    // ...

    &--primary {
      // ...
    }
  }
}

    # 格式化

    格式化部分我们基本都可以使用工具来解决，但这里我还是希望大家能看看，并养成开发时就进行格式化书写的习惯。

    1. 选择器与 { 之间留一个空格
    2. 结束的 } 单独占一行
    3. : 与后面的属性值之间保持一个空格，:与前面的属性名之间不留空格
    4. 多个选择器在一起时，每个选择器单独占一行
